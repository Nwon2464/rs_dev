# 붉은보석 데미지 합산 구조 역추적

## 프로젝트 목표

로컬 게임 데이터와 읽기 전용 관찰 자료를 바탕으로 데미지 관련 Capa를
효과 버킷별로 분리하고, 최종 산식에 필요한 증거를 축적한다. 이 프로젝트는
현재 단계에서 게임과 동일한 최종 데미지를 계산하거나 산식을 단정하지 않는다.

조사 산출물은 다음 순서로 남긴다.

1. 원시 근거는 `data/evidence/`에 기록한다.
2. Capa와 버킷의 작업 초안은 `notes/`에 기록한다.
3. 재현 가능한 가설 검증 결과는 `data/experiments/`에 기록한다.
4. 파생·정규화 결과는 `data/processed/`에 기록한다.

## 확정·추론·미확정 구분 원칙

모든 주장과 근거 행은 아래 세 근거 수준 중 하나를 사용한다.

- `confirmed`: 로컬 원본 파일, 실행 메모리, 또는 재현 가능한 관찰에서 값·식별자·직접 연결을 확인했다. 확인한 사실의 범위를 넘는 계산 순서는 포함하지 않는다.
- `inferred`: 둘 이상의 `confirmed` 근거로 가장 잘 설명되는 구조다. 반증 가능성과 추론 근거를 함께 기록하며, 확정된 계산식처럼 사용하지 않는다.
- `unknown`: 근거가 없거나 서로 충돌해 아직 판정할 수 없다. 임의의 기본값·합산 순서·상한을 채워 넣지 않는다.

같은 표시 문구라도 Capa ID 또는 내부 효과 타입이 다르면 먼저 별도 버킷으로
보관한다. 버킷 병합, 곱셈 순서, 제한 적용 시점, 정수 처리 방식은 직접 근거가
생기기 전까지 `inferred` 또는 `unknown`으로 남긴다.

이 문서는 로컬 `C:\game\Red Stone` 설치본에서 장비·스킬·특수 효과의 데미지 수치가 어디에 저장되고, 클라이언트가 이를 어떤 단위로 구분하는지 역추적한 기록이다.

- 조사 기준일: 2026-07-11
- 게임 경로: `C:\game\Red Stone` (`/mnt/c/game/Red Stone`)
- 기존 프로젝트 코드와 산출물은 수정하지 않았다.
- 게임 파일도 읽기 전용으로 조사했다.
- 저장소의 조사 자료는 `damage_experiment/` 아래에만 추가한다.
- 실제 공격 패킷 변조, 프로세스 후킹, 게임 파일 변경은 수행하지 않았다.

## 현재 결론

`Data` 폴더에는 데미지 계산의 **입력값과 효과 정의**가 있다. 그러나 여러 효과를 최종 데미지로 결합하는 완전한 산식은 독립된 DAT나 스크립트로 존재하지 않는다.

실행 중 해제된 구버전 클라이언트 메모리를 조사한 결과, 데미지 관련 효과가 다음처럼 별도 내부 타입으로 분리되어 있었다.

- 기본 물리 대미지 증가
- 기본 마법 대미지 증가
- 물리 치명타 대미지
- 물리 더블 크리티컬 대미지
- 물리 강타 대미지
- 마법 치명타 대미지
- 마법 강타 대미지
- 최종 물리 공격력
- 최종 마법 공격력
- 최종 물리 대미지
- 최종 마법 대미지
- 물리·마법 공통 최종 대미지
- 물리·마법 한계 대미지

따라서 화면 문구에 모두 `대미지 증가`가 들어가더라도 전부 한 합계로 더하면 안 된다. 최소한 **효과 종류별 버킷 합산 → 버킷별 제한 적용 → 서로 다른 계산 단계에서 반영**하는 구조다.

현재까지 확정할 수 없는 것은 버킷 사이의 정확한 곱연산 순서, 방어·저항 적용 위치, 한계 대미지 처리 시점, 정수 절삭 시점이다. 이 부분은 클라이언트의 전투 표시 루틴이나 서버 계산을 추가로 추적해야 한다.

## 조사 자료와 버전 차이

### 현재 설치본

`RedStone.ini`에서 현재 클라이언트 버전은 `884`로 확인된다.

현재 `Red Stone.exe` SHA-256:

```text
6481966a25daf61d0d75454e8422cd40463a5de4410561240eb320968bf1093b
```

현재 실행 파일은 PE32지만 일반적인 섹션 배치가 아니다.

|---|---:|---:|---|
| `TEXT` | `0x0A671000` | `0` | 실행 시 복원되는 가상 코드 영역 |

진입점도 큰 가상 영역 뒤쪽을 가리킨다. 디스크의 `Red Stone.exe`에서 평문 함수명이나 DAT 참조가 거의 나오지 않는 이유는 코드가 보호·패킹되어 있기 때문이다.

오류 보고서에 포함된 개인 식별 정보는 이 문서에 옮기지 않았다.

## `capa.dat`: 데미지 효과의 사전

`capa.dat` 레코드는 텍스트 앞에 8개의 32비트 필드를 가진다. 현재 직접 확인된 안정적인 항목은 다음과 같다.

```text
[효과 ID, 분류, 제한 후보값, 필드3, 필드4, 필드5, 필드6, 필드7]
```

- ID 939: 물리 치명타 대미지 증가량을 더블 크리티컬 대미지 증가로 변환
- ID 943: 마법 치명타 확률 100% 초과분에 비례해 마법 증폭 한계 증가
- ID 944: 마법 치명타 대미지를 마법 강타 대미지로 변환
- ID 948: 마법 대미지 증가량에 비례해 대상 최종 방어력 감소
- ID 1000: 소모 CP에 비례한 최종 대미지 증가와 최대치
- ID 1053: 특정 상태이상 대상에 대한 최종 대미지 증가

이 규칙들은 전체 데미지 공식이 아니라 조건부 효과 하나를 계산하기 위한 파라미터다.

## `VariableCapa.dat`의 실제 성격

처음에는 `VariableCapa.dat`가 능력치 상한이나 합산 테이블일 가능성을 검토했다. 실행 메모리의 로더와 UI 문자열을 교차검증한 결과, 일반 데미지 합산 테이블로 볼 수 없다.

직접 확인된 실행 문자열:

```text
cVariableCapaMgr_cm::load
ChangeVariableCapa
VariableCapaRerollTest
error in %s - out of range variable capa option count [%d]
error in %s - out of range variable capa table count [%d]
```

로더는 파일 매직 `0xABC123EF`를 검사하고, 옵션 개수 최대 7개와 테이블 개수 최대 64개를 검사한다. `ChangeVariableCapa` UI 및 `RerollTest`와의 연결을 고려하면 가변 옵션의 변경·재추첨 후보표다.

따라서 `VariableCapa.dat`의 Capa ID와 수치 행을 일반 캐릭터 데미지 합산 상한으로 사용하면 안 된다.

## `AssaultChallenge.dat`의 `CorrectFinalDamage`

전체 데이터에서 `CorrectFinalDamage`라는 명칭은 `AssaultChallenge.dat`에서만 발견됐다.

```text
CorrectFinalDamageRG_P000
Step
StepStartHP_level
TitleIconImgIdx
TitleName
```

`CorrectFinalDamageRG_P000`은 50회 반복되며 도전 단계, 시작 HP 단계, 칭호 정보와 함께 저장되어 있다. 일반 캐릭터의 장비 데미지를 합산하는 공식이 아니라 Assault Challenge 콘텐츠의 단계별 보정 데이터다.

| 66 | `cOSEIncLastMgAtkP_cm` | 최종 단계 마법 공격력 증가율 후보 |
| 67 | `cOSEIncPhLimitDmgF_cm` | 물리 한계 대미지 고정 증가 |
| 68 | `cOSEIncMagicLimitDmgF_cm` | 마법 한계 대미지 고정 증가 |
| 72 | `cOSEIncFinalPhDamagePerF_cm` | 최종 물리 대미지 증가 |
| 73 | `cOSEIncFinalMagicDamagePerF_cm` | 최종 마법 대미지 증가 |
| 74 | `cOSEIncFinalDamagePerF_cm` | 물리·마법 공통 최종 대미지 증가 |

`P`, `F`, `PerF` 접미사의 정확한 원 개발 명칭은 남아 있지 않다. 코드 사용 형태상 `P`는 비율 계열, `F`는 고정 또는 부동소수 계열로 보이지만 확정 명칭으로 사용하지 않는다.

### 효과 팩토리 교차검증

내부 타입 생성자는 하나의 공통 팩토리에서 호출된다. 구버전 코드 `0x009B4100` 이후에는 각 타입마다 동일하게 `0x6C`바이트 객체를 할당하고 대응 생성자를 호출한다.

특히 최종 데미지 타입은 팩토리에서도 연속된 세 분기로 나타난다.

| 팩토리 호출 주소 | 생성자 | 객체 내부 타입 |
|---:|---:|---:|
| `0x009B4756` | `0x00A53080` | 72, 최종 물리 |
| `0x009B478B` | `0x00A52DF0` | 73, 최종 마법 |
| `0x009B47C0` | `0x00A52B60` | 74, 공통 최종 |

따라서 세 클래스가 우연히 남은 이름만 다른 것이 아니라, 런타임 효과 팩토리에서 서로 다른 타입으로 실제 생성되는 것을 확인했다.

## 효과 파라미터 전개 루틴

세 기본 최종 데미지 클래스의 가상 함수 구조는 거의 동일하다.

| 클래스 | 주요 계산 함수 주소(0.0859) |
|---|---:|
| 공통 최종 대미지 | `0x00A52C30` |
| 최종 마법 대미지 | `0x00A52EC0` |
| 최종 물리 대미지 | `0x00A53150` |

공통 함수는 효과 데이터의 세 부동소수 값과 16비트 단계값을 읽는다. x87 연산 순서를 해석하면 다음 형태가 강하게 나타난다.

```text
전개값 = min(기본값 + 단계증가값 × 단계수, 최대값)
```

그 후 전개값이 유효 범위인지 검사하고 출력 파라미터에 기록한다. 동일 객체가 3개의 파라미터 슬롯을 유지하며, 로드 시 세 정수 파라미터를 각각 부동소수 값으로 변환한다.

이 식은 **여러 장비 옵션의 합산식이 아니다.** 스킬 레벨이나 효과 단계에 따라 개별 OSE 파라미터 하나를 전개하는 함수다. `단계수`의 정확한 게임 의미도 호출자별로 다를 수 있으므로 아직 확정하지 않는다.

## Capa 저장 구조

구버전의 `cPlayerCapas_cm`은 초기화할 때 다음 구조를 만든다.

- 고정 영역 `0x348`바이트를 0으로 초기화한다.
- 이는 32비트 값 210개에 해당한다.
- 별도로 `cPlayerUncommonCapas_cm` 컨테이너와 여러 트리형 컨테이너를 유지한다.
- 일부 외부 Capa ID는 하드코딩된 매핑 함수를 거쳐 내부 1~31번 특수 타입으로 변환된다.

관련 코드 주소(0.0859):

| 주소 | 관찰 내용 |
|---:|---|
| `0x009F6E20` | `cPlayerCapas_cm`의 0x348바이트 고정 영역 초기화 |
| `0x009F6EF0` | 외부 Capa ID 범위 검사 후 특수 Capa 조회 |
| `0x009F1430` | 일부 외부 Capa ID를 내부 특수 타입으로 변환 |
| `0x009F6C90` | 내부 특수 타입 컨테이너 조회 |

이는 클라이언트가 효과를 받을 때마다 문구를 다시 해석하는 것이 아니라, 정규 능력치 영역과 특수 효과 컨테이너에 정리해 보관한다는 증거다.

다만 210개 슬롯 각각의 의미와 장비·버프·패시브 사이 합산 우선순위는 아직 복구하지 못했다.

## 현재까지 복구된 합산 구조

현재 증거로 안전하게 표현할 수 있는 구조는 다음과 같다.

```text
item.dat / 장비 옵션 / InstandardEquip.dat
                    │
                    ├─ Capa ID와 수치
                    │
                    ▼
              capa.dat 정의 조회
                    │
                    ├─ 효과 분류
                    ├─ 제한 후보값
                    └─ 표시 및 특수 규칙
                    │
                    ▼
        PlayerCapas / 특수 OSE 타입별 저장
                    │
       ┌────────────┼─────────────┐
       ▼            ▼             ▼
  기본 물리/마법  치명타·강타   최종 물리/마법/공통
       │            │             │
       └────────────┴─────────────┘
                    │
                    ▼
        클라이언트 전투 표시 또는 서버 계산
```

### 구현에 사용할 수 있는 임시 원칙

향후 계산기를 만들 때 데이터가 더 확인되기 전까지 적용할 수 있는 보수적인 원칙이다.

1. 동일 Capa ID의 수치는 먼저 합산한다.
2. 의미가 비슷하더라도 Capa ID와 내부 효과 타입이 다르면 별도 버킷으로 유지한다.
3. `capa.dat` 세 번째 필드는 검증 전까지 `상한 후보`로만 저장한다.
4. 기본 물리·기본 마법·치명타·더블 크리티컬·강타·최종 공격력·최종 데미지·한계 데미지를 서로 합치지 않는다.
5. 조건부 효과는 조건을 만족할 때만 대응 단계 버킷에 추가한다.
6. 정확한 곱셈 순서와 절삭 방식이 확인되기 전에는 `게임과 동일한 최종 데미지`라고 표시하지 않는다.

이 원칙은 원시 옵션 집계에는 사용할 수 있지만 완전한 전투 데미지 계산식은 아니다.

## 확정·추론·미확정 구분

### 확정

- `Data`에는 Capa ID와 개별 효과 수치가 존재한다.
- `capa.dat`에는 물리, 마법, 치명타, 강타, 최종 물리, 최종 마법, 공통 최종 데미지가 별도 ID로 존재한다.
- 실행 메모리에도 이 효과들이 별도 C++ 클래스로 존재한다.
- 공통·물리·마법 최종 데미지는 내부 타입 74·72·73으로 분리된다.
- `VariableCapa.dat`는 일반 데미지 합산 테이블이 아니라 옵션 변경·재추첨 계열이다.
- `AssaultChallenge.dat`의 `CorrectFinalDamage`는 콘텐츠 전용 단계 데이터다.
- 디스크의 현재 실행 파일은 보호·패킹되어 있다.

### 강한 추론

- `capa.dat`의 세 번째 32비트 필드는 효과별 누적 또는 적용 상한이다.
- 개별 OSE의 세 값은 기본값, 단계 증가량, 최대값으로 사용된다.
- 개별 OSE 전개식은 `min(기본값 + 증가량 × 단계수, 최대값)` 형태다.
- 동일 단계의 같은 효과는 먼저 누적되고, 서로 다른 OSE 타입은 이후 계산 단계에서 따로 적용된다.

### 미확정

- 서로 다른 Capa ID가 같은 내부 누적 슬롯을 공유하는 전체 매핑
- 장비, 버프, 패시브, 스킬 효과 사이의 합산 우선순위
- 물리·마법 기본 대미지 공식
- 치명타·강타·더블 크리티컬의 정확한 적용 순서
- 방어력과 속성 저항 적용 순서
- 최종 공격력과 최종 데미지 사이의 순서
- 한계 대미지와 초과 적용률 계산 순서
- 각 단계의 정수 변환, 반올림, 버림 방식
- 실제 전투 데미지를 클라이언트와 서버 중 어디까지 계산하는지
- 0.0859에서 확인한 내부 타입 번호가 현재 0.0884에도 동일한지
