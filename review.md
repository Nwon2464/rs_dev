# 비규격 장비 옵션 데이터·렌더링 리뷰

## 종합 판정

현재 `C:\game\Red Stone\Data\InstandardEquip.dat`에서 다시 추출한 결과는 저장된 `data/processed/instandard_equipment.json` 및 CSV와 바이트 단위로 일치한다. 현재 원본 기준 데이터는 최신이며, 장비별 옵션 ID 조인도 정상이다.

- 장비군: 34개
- 옵션 정의: 80개
- 원본 `OptionsByItemType`에 직접 연결된 옵션: 78개
- 로컬 교차검증으로 목걸이에 보완 배정한 옵션: ID 922, 1045
- 화면에 표시하는 전체 옵션 정의: 80개
- 옵션별 원본 Tier: 10개, 각 Tier의 수치 후보: 10개
- 활성 수치 행: 7,640개
- 비활성 센티널 행: 360개
- CSV 전체 행: 8,000개
- 장비별 렌더링 직전 CSV: 8,731개 (`장비군 × 옵션 × 공통 OptionLevel 구간`)

데이터 스키마는 버전 2로 보강했다. 원본 로컬 순번인 `raw_tier_index`와 전 옵션 공통 정렬축인 `option_level_group_index`를 분리했으며, 원본 파일 크기와 SHA-256도 JSON에 저장한다. 통합 HTML은 렌더링 직전 CSV를 입력으로 사용하고 공통 정렬축을 `공통 수치 구간`으로 표시한다.

ID 1045는 `item.dat`에서 마도풍 계열 목걸이 5종의 고정효과로, ID 922는 `item_option_open.dat`에서 목걸이 대상 옵션으로 확인했다. 두 ID가 공유하는 로컬 장비군 8 연결을 근거로 목걸이에 보완 배정했으며, 화면과 CSV에서는 `supplemental_local_cross_reference`로 원본 직접 배정과 구분한다.

통합 HTML에는 개방 옵션 데이터와 비규격 데이터가 모두 내장되어 있다. 비규격 화면의 장비 선택, 태그 필터, 공통 수치 구간 필터와 수치 치환은 정적 검증을 통과했다. 실제 브라우저 클릭·레이아웃 자동화는 아직 없다.

## 운영 구조 추가 확인

외부 운영 참고 자료 `https://sokomin.github.io/information/nonstandard-equipment.html`와 1500 장갑 실물 사례를 대조해 다음을 추가로 확인했다.

- 1250·1500·1750은 베이스 장비의 요구 레벨이다. 비규격 옵션 Tier 0~9와 같은 축이 아니다.
- 일반 등급은 비규격 옵션 0개, DX는 1~2개, ULT는 3~5개이며 최대 5칸이다.
- 외부 운영 자료는 수치 재설정이 동일 Tier 안에서 이루어진다고 설명한다. 다만 로컬 DAT에는 추첨 순서나 게임 화면의 티어 표시를 직접 연결하는 레코드가 없다.
- 접두 문구는 표시 이름이며 옵션 추첨 내용을 직접 나타내지 않는다.
- 요구 레벨 1500이 모든 옵션에 하나의 공통 `TierData` 그룹을 강제하지 않는 것은 실물 사례로 확인된다.

참고 사이트의 비규격 베이스 장비는 204개이고 로컬 후보 장비군은 34개다. `204 = 34 × 6`이며, 사이트 표는 각 장비군에 두 베이스 계열과 요구 레벨 1250·1500·1750의 세 변형을 보여준다. 따라서 현재 34개 `OptionsByItemType` 행은 개별 아이템 204개를 직접 나열한 것이 아니라, 같은 부위/무기군의 6개 베이스가 공유하는 옵션 후보표로 보는 것이 유력하다. 이는 로컬 item ID 204개를 직접 조인한 결과는 아니므로 ‘강한 구조 추론’으로 분류한다.

1500 장갑 실물 수치는 로컬 원본과 다음처럼 일치한다.

| 실물 옵션 | 값 | 로컬 옵션 ID | 원본 TierData 그룹 | 그룹 내 위치 |
|---|---:|---:|---:|---:|
| 물리 공격력 증가 | 95% | 460 | 1 | 5번째 |
| 무기 최대 공격력 | 17 | 722 | 5 | 4번째 값부터 반복 |
| 물리 강타 대미지 | 46% | 755 | 4 | 1번째 값부터 반복 |
| 물리 강타 확률 | 32% | 462 | 3 | 8번째 값부터 반복 |
| 공격 속도 | 34% | 74 | 2 | 7번째 값부터 반복 |

따라서 원본 수치가 틀린 것은 아니다. 기존 화면의 `Tier` 명칭은 옵션 내부 인덱스를 전역 단계처럼 사용한 것이 문제였다. 수정 화면은 게임의 공식 티어라고 단정하지 않고 `공통 수치 구간`으로 표시한다.

### TierData에 대해 확정된 것과 미확정인 것

| 항목 | 상태 | 근거 |
|---|---|---|
| 옵션마다 `TierData` 배열이 존재 | 확정 | MessagePack 필드명 |
| 각 그룹에 `Tier`, `OptionLevel`, 10개 `TierValue` 존재 | 확정 | 로컬 구조 파싱 |
| 현재 `Tier` 값이 0~9 | 확정 | 전체 80개 정의 검사 |
| 실물 수치가 해당 그룹의 10개 값 안에 존재 | 확정 | 1500 장갑 5개 옵션 역추적 |
| 1250/1500/1750이 그룹 0/1/2에 대응 | 부정 | 1500 실물 값이 그룹 1·5·4·3·2에 분산 |
| 게임 화면이 이 숫자를 옵션 Tier로 표시 | 미확정 | 로컬 UI 연결 문구 없음 |
| 먼저 Tier를 추첨한 뒤 10개 값을 추첨 | 미확정 | 실행 로직/확률 테이블 없음 |

| 수치 재설정이 같은 그룹 안에서만 동작 | 외부 자료상 유력 | 참고 사이트 설명, 로컬 직접 근거 없음 |

### Tier 필드와 OptionLevel의 실제 정렬 관계

활성 `OptionLevel`은 정확히 다음 10개 공통 키로 수렴한다.

```text
1000, 5090, 17310, 27500, 28050, 30500, 32500, 36000, 40000, 40990
```

옵션 정의는 세 형태뿐이다.

| 정의 형태 | 옵션 수 | 로컬 Tier | OptionLevel 키 |
|---|---:|---|---|
| 전체 구간 | 69 | 0~9 | 공통 키 0~9 전체 |
| 고급 구간 전용 | 10 | 0~6 | 공통 키 3~9 |
| 저급 구간 전용 | 1 | 0~3 | 공통 키 0~3 |

고급 구간 전용 옵션에서 로컬 `Tier=0`은 `OptionLevel=27500`, 즉 공통 키 정렬 순번 3이다. 로컬 `Tier`는 전 옵션에 공통된 품질 단계가 아니라 옵션별 배열 인덱스다. 현재 구조에서 전 옵션을 같은 축으로 정렬하는 안정적인 키는 `OptionLevel`이다. 다만 `OptionLevel` 숫자의 의미가 누적 확률 경계인지, 품질 점수인지, 다른 내부 값인지는 아직 미확정이다.

영향받는 장비 연결 옵션은 다음 8개다.

| ID | 옵션 | 현재 로컬 Tier 범위 | 공통 OptionLevel 순번 |
|---:|---|---|---|
| 464 | 물리 더블 크리티컬 대미지 | 0~6 | 3~9 |
| 752 | 마법 치명타 대미지 | 0~6 | 3~9 |
| 723 | 마법 강타 확률 | 0~6 | 3~9 |
| 754 | 마법 강타 대미지 | 0~6 | 3~9 |
| 703 | 물리 한계 대미지 | 0~6 | 3~9 |
| 704 | 물리 한계 초과 적용률 | 0~6 | 3~9 |
| 705 | 마법 한계 대미지 | 0~6 | 3~9 |
| 706 | 마법 한계 초과 적용률 | 0~6 | 3~9 |

목걸이에 보완 배정한 922·1045도 같은 3~9 구조다. 수정 렌더러는 `OptionLevel → 공통 순번`을 사용하므로 고급 옵션 10개가 공통 구간 3~9에 정렬된다.

### 렌더링 직전 CSV 구조

`data/processed/instandard_equipment_render_rows.csv`는 활성 데이터만 장비별로 평탄화한 8,731행이다. 이 CSV는 화면 레이아웃을 결정하지 않으며, 구조 검증과 향후 렌더러 입력을 위한 중간 계층이다.

| 필드 | 의미 |
|---|---|
| `item_group_id/name` | 장비군 직접 조인 결과 |
| `option_order_in_equipment` | `OptionsByItemType` 원본 배열 순서 |
| `option_id/name/template/tags` | capa 및 OptionData 조인 결과 |
| `raw_tier_index` | 해당 옵션 내부 `Tier` 원문 |
| `option_level_raw` | 원본 `OptionLevel` 공통 키 |
| `option_level_group_index` | 활성 공통 키 정렬 순번, 0 기반 |
| `option_level_group_number` | 같은 순번의 1 기반 표시용 값 |
| `group_index_offset_from_raw_tier` | 공통 순번과 로컬 인덱스 차이 |
| `value_0_min/max` | 10개 후보의 실제 최소·최대 |
| `roll_01_raw`~`roll_10_raw` | `[value0,value1,value2]` 원시 벡터 |
| `roll_01_display`~`roll_10_display` | capa 표시문에 벡터를 대입한 값 |
| `mapping_basis` | 행 생성에 사용한 로컬 테이블 |

`option_level_group_index`는 구조상 안정적인 공통 정렬 순번이지만 게임의 공식 티어 명칭으로 간주하지 않는다.

구조 검증 명령은 `python3 scripts/validate_instandard_equipment.py`다. React 앱은 공개 데이터 파일을 fetch하고 Vite 빌드로 검증한다.

## 검증 범위와 방법

1. 원본 DAT에서 `/tmp`로 JSON·CSV·Markdown을 다시 생성했다.
2. 임시 JSON과 저장된 JSON, 임시 CSV와 저장된 CSV를 각각 `cmp`로 비교했다.
3. `OptionsByItemType`의 모든 장비군과 옵션 ID를 `OptionData`, `capa.dat`, `simpleGameText.dat` 조인 결과에 역검증했다.
4. 장비별 옵션 ID 중복, 미정의 ID, `selectable` 플래그 누락을 검사했다.
5. 활성 Tier 수, 센티널 수, 수치 정렬 상태, 표시문 `[0]`, `[0%]`, `[0％]`, `[0.1%]` 치환 범위를 검사했다.
6. 통합 HTML의 두 내장 페이지와 진입 링크를 파싱해 데이터가 실제 번들에 포함됐는지 확인했다.

현재 실행 환경에는 Node/브라우저 자동화 런타임이 없어 실제 클릭·레이아웃을 자동 브라우저로 재생하지는 못했다. DOM 데이터 내장과 렌더링 함수의 입력 범위는 정적 검증했다.

| 7 | 10 | 나머지 3개는 `OptionLevel=99999` 센티널 |
| 4 | 1 | 나머지 6개는 센티널 |

원본 CSV는 센티널 360행도 포함하고 `enabled=False`로 표시한다. 렌더링 직전 CSV는 활성 행만 포함하므로 화면에는 센티널이 나오지 않는다.

## 렌더링 확인

### 정상인 부분

- React 앱은 공개 개방 옵션 8,008행과 비규격 장비군 34개·옵션 정의 80개를 fetch한다.
- 비규격 화면은 선택한 장비의 `option_ids`만 `optionMap`에서 읽으므로 다른 장비 옵션이 섞이지 않는다.
- 장비 변경 시 태그는 전체, Tier는 0으로 초기화되어 이전 장비의 유효하지 않은 필터가 남지 않는다.
- 태그는 옵션의 `TagName` 배열과 정확히 비교한다.
- `OptionLevel=99999`인 Tier는 `tiers(option)`에서 제외된다.
- ASCII `%`와 전각 `％`, 소수 정밀도 표기 `[0.1%]`를 치환한다.
- 전체 보기와 구간 보기는 렌더링 직전 CSV의 실제 최소·최대 및 10개 원시 벡터를 사용한다.
- 옵션 행을 누르면 선택 Tier의 수치 10개가 모두 표시된다.

### 원본 직접 배정과 보완 배정의 표시

옵션 ID 922와 1045는 원본 `OptionsByItemType`에는 없지만 다른 로컬 테이블에서 목걸이 연결이 확인된다. 현재 화면은 두 옵션을 목걸이에 표시하면서 `로컬 교차검증 보완` 배지를 붙여 원본 직접 배정과 혼동되지 않게 한다.

## 발견 사항과 보완점

### P1 · 생성 파이프라인에 최신 원본 검증이 없다

독립 Viewer 생성기는 이미 생성된 JSON과 CSV만 읽는다. 원본 DAT가 바뀌어도 수집기를 먼저 실행하지 않으면 오래된 데이터가 정상인 것처럼 다시 번들된다.

권장 조치:

1. `collect_instandard_equipment.py` → 개방 옵션 수집기 → 두 JSON/CSV 검증 → 독립 HTML 생성 순서를 하나의 빌드 명령으로 묶는다.
2. JSON의 `source`에 원본 파일별 SHA-256, 크기, 수정 시각을 저장한다.
3. 독립 HTML에도 사용한 리비전과 원본 해시를 표시한다.
4. processed 파일의 해시가 현재 원본에서 재생성한 값과 다르면 HTML 생성을 실패시킨다.

### 보완 완료 · 원본 미배정 옵션의 로컬 교차검증

현재 장비 화면은 원본 `OptionsByItemType` 연결 78개와 목걸이 보완 옵션 2개를 함께 표시한다. 922·1045에는 근거 배지를 붙였고 JSON/CSV에도 원본 직접 배정과 보완 배정을 분리해 저장한다.

권장 조치:

- 향후 원본 후보표에 922·1045가 직접 추가되면 보완 규칙을 자동 제거하거나 직접 배정으로 승격한다.
- `raw_selectable_option_count`, `supplemental_option_ids`, `unused_option_ids`를 함께 진단한다.
- 새로운 원본 미배정 옵션이 생기면 빌드 경고를 출력한다.

### 보완 완료 · 렌더러의 공통 정렬축

기존 렌더러는 `tier.tier`를 직접 필터와 라벨에 사용했다. 수정 렌더러는 `option_level_group_index`를 사용하며 명칭을 `공통 수치 구간`으로 제한한다.

옵션 464의 로컬 0→공통 3, 로컬 6→공통 9 변환과 같은 규칙을 정적 HTML 검증에 고정했다.

### 데이터 계층 보완 완료 · value1/value2 원시값 보존

기존 수집기는 value1/value2가 0이 아니면 실패했다. 현재는 세 값을 모두 JSON·원본 CSV·렌더링 직전 CSV에 보존한다. 렌더링 직전 표시문 변환기도 `[0]`, `[1]`, `[2]`와 소수 자릿수를 처리한다.

HTML의 JavaScript 포매터도 `[0]`, `[1]`, `[2]`, 소수 자릿수, ASCII/전각 퍼센트를 처리한다.

### 보완 완료 · 중복 통합 Viewer 제거

iframe으로 독립 Viewer를 다시 감싸던 통합 HTML과 생성기를 제거했다. React 이관 전에는 두 독립 Viewer만 유지한다.

### 데이터 계층 보완 완료 · 수치 범위의 정렬 가정 제거

렌더링 직전 CSV는 10개 value0의 실제 최솟값·최댓값을 계산하고 원본 순서를 별도 열에 보존한다. HTML도 이 최소·최대 및 원시 순서를 사용한다.

추가 권장 조치는 수집 단계에서 배열 정렬 여부를 검증하고 변경 시 경고하는 것이다.

### P2 · 센티널 판정 규칙이 필드 하나에만 의존한다

`scripts/collect_instandard_equipment.py:242`는 `OptionLevel != 99999`만으로 활성 여부를 정한다. 현재 패턴에서는 타당하지만 규칙의 근거가 데이터 스키마나 테스트로 고정돼 있지 않다.

권장 조치:

- `OptionLevel=99999`일 때 value0도 999/99999 센티널 패턴인지 함께 검사한다.
- 패턴이 달라지면 자동 제외하지 말고 빌드를 실패시켜 검토하도록 한다.
- JSON에 `enabled_basis: "option_level_sentinel"` 같은 판정 근거를 남긴다.

### P2 · 브라우저 회귀 테스트가 없다

데이터와 HTML 문자열은 검증되지만 실제 클릭, 필터 변경, 상세 패널, 모바일 레이아웃은 자동 확인하지 않는다.

권장 최소 테스트:

1. 헬멧 선택 시 Tier 0 후보 22개 표시.
2. 관 선택 시 후보 33개 중 Tier 7에서 센티널 옵션 제외 확인.
3. 목걸이에서 `스킬 쿨타임 감소`가 Tier 0~3만 표시.
4. 장비 변경 후 태그와 Tier 초기화 확인.
5. `%`, `％`, 소수형 표시문 수치 치환 확인.
6. 상세 패널에 정확히 10개의 수치 셀 표시.
7. 개방 옵션 탭 ↔ 비규격 탭 전환과 URL hash 복원 확인.

### P3 · CSV의 기본 사용 시 센티널 오해 가능성이 있다

CSV 8,000행 중 360행은 비활성이다. `enabled` 컬럼이 있지만 단순 통계 도구에서 필터하지 않으면 999/99999가 실제 최고 수치처럼 집계될 수 있다.

권장 조치:

- 원본 보존 CSV는 현재처럼 유지한다.
- 별도로 `instandard_equipment_tiers_active.csv`를 생성하거나 파일 상단 문서에 `enabled=True` 필터를 명시한다.

### 부분 보완 · 출처 메타데이터와 스키마 버전

JSON에 `schema_version=2`, 생성기 경로, 원본 파일별 크기와 SHA-256을 추가했다. 리비전과 원본 경로도 계속 보존한다.

권장 조치:

- 남은 항목으로 `generated_at`과 생성기 자체 버전/해시를 추가한다.
- 독립 HTML 하단에 같은 값을 표시한다.
- 생성 결과와 검증 요약을 별도 manifest JSON으로 둔다.

## 권장 적용 순서

1. 새 CSV와 JSON을 대상으로 구조 검증을 더 축적한다.
2. 단일 빌드 명령과 원본 해시 불일치 실패 처리를 추가한다.
3. 보완 배정과 원본 직접 배정의 차이를 계속 검증한다.
4. 활성 센티널 규칙을 검증 가능한 스키마로 만든다.
5. iframe 통합을 단일 DOM 앱으로 리팩터링한다.
6. 실제 브라우저 회귀 테스트를 추가한다.

## 최종 결론

현재 원본 후보 78개와 로컬 교차검증으로 목걸이에 보완한 922·1045를 구분해 저장·표시한다. 저장 데이터는 최신 원본과 일치하고 현재 데이터 범위의 렌더링 값도 정확하다. 가장 큰 관리 위험은 원본 갱신 후 수집기를 다시 실행하지 않는 상태와, 향후 원본 직접 배정이 생겼는데 기존 보완 규칙이 중복되는 상태다.
